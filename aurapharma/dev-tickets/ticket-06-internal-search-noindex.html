Issue

The site exposes a public internal search at /website/search. Both the bare endpoint and every parameter URL it produces (?search=<query>&order=<sort>) are currently indexable. The parameter URLs do correctly canonicalise to /website/search, which consolidates the parameter combinations to a single canonical target. But that target itself has no <meta name="robots"> tag, so it inherits the default index,follow directive. The page renders empty results scaffolding (0 H1, 0 H2, 0 H3, no description, 14 H5s used decoratively), so what Google would index is a thin-content page with no value.

Verified evidence

$ curl -s "https://www.aurapharma.org/website/search?search=test&order=name+asc" | grep -iE "canonical|robots|title"
<link rel="canonical" href="https://www.aurapharma.org/website/search"/>
<title>Any Search Results | aurapharma</title>

$ curl -s "https://www.aurapharma.org/website/search" | grep -iE "canonical|robots"
<link rel="canonical" href="https://www.aurapharma.org/website/search"/>

Two confirmations: (1) no <meta name="robots"> on either URL, so both are indexable by default; (2) no Disallow: /website/search in robots.txt.

Why this is wrong

Google has been explicit on this since 2007. Search Central docs and repeated guidance from John Mueller spell out four reasons internal search results should never be indexable:

  1. Search-results-within-search-results is poor UX. A user clicks a Google result and lands on another search page they have to re-search through. Google's Search Quality Rater Guidelines mark this as low-quality.
  2. Thin content risk. Empty or low-result search pages can trigger the "Thin content with little or no added value" guideline and have caused manual actions in the past.
  3. Crawl-budget waste. Search query URLs are infinite. Without a noindex, every parameter combination becomes a candidate URL, eating budget that should go to product and category pages.
  4. User-generated query risk. Crawlers can index URLs containing whatever a visitor typed. Imagine search queries with profanity, competitor names, or spam appearing as URLs in site:aurapharma.org results.

Fix

Step 1: Add noindex to the search template

Edit the search-results template so every response includes:

<meta name="robots" content="noindex,follow">

The ,follow portion is intentional. It tells Google not to index the page itself but to still follow internal links from it (so any product links shown in search results still pass equity).

Do not use noindex,nofollow. The nofollow would block link discovery from the search page, which is unnecessary and slightly harmful.

Step 2: Wait, then optionally add a Disallow

Wait roughly two weeks after the noindex ships so Googlebot can recrawl and process the directive. After that, you can add to robots.txt:

Disallow: /website/search

This saves crawl budget on future visits. The order matters: shipping the Disallow first would block Google from ever seeing the noindex tag, leaving any already-indexed search URLs stuck in the index without a removal directive.

Step 3 (optional): Stop linking to search URLs internally

Audit the templates and remove any internal links that point to /website/search?search=... with hardcoded parameters. Search results should be reachable via the search input, not via persistent links from navigation or content.

Verification

  • Run curl -s "https://www.aurapharma.org/website/search?search=test" | grep -i robots. Expected: <meta name="robots" content="noindex,follow">.
  • Run the same on the bare /website/search endpoint. Expected: same noindex tag.
  • Confirm parameter URLs still canonicalise to /website/search (the canonical behaviour should not regress).
  • Once Google Search Console is connected, monitor Pages > Not indexed for the count of "Excluded by 'noindex' tag" entries under /website/search. The count should rise as Google recrawls.
  • Two weeks after deploy, search site:aurapharma.org/website/search in Google. Expected: zero or rapidly diminishing results.
  • After step 5 confirms removal, ship the Disallow: /website/search robots.txt line and re-verify with curl -s https://www.aurapharma.org/robots.txt.

Why this is critical

This is a preventive fix more than a corrective one. The site is new, so the volume of search URLs already in Google's index is likely small. But the longer it stays open, the more parameter combinations get crawled, and the harder cleanup becomes. The fix is one line in one template. There is no reason to delay it.

This is also why structured data (AURA-005) and the homepage canonical fix (AURA-001) compound: every fix that makes Google understand the site better also makes the wrong things easier to find. Closing the indexable-search door now means that work pays off cleanly.
