What's actually on the site today

This ticket previously claimed the site had zero structured data. That was wrong. The site has structured data, but in microdata format (Schema.org expressed via HTML attributes), not JSON-LD. The microdata is partial and the schema chosen for the shop page is not the right type. Below is the verified state.

Sitewide protocol issue: the existing microdata uses itemtype="http://schema.org/..." (HTTP, not HTTPS). Both still work, but https://schema.org/... is current best practice and the form Google's documentation uses.

Verified evidence

Product page (/shop/vitamin-c-6)

$ curl -s "https://www.aurapharma.org/shop/vitamin-c-6" \
    | grep -oiE 'itemtype="[^"]+"|itemprop="[^"]+"' | sort | uniq -c

   4 itemscope
   1 itemtype="http://schema.org/Product"
   1 itemtype="http://schema.org/Offer"
   1 itemprop="name"          (value: "Vitamin C")
   1 itemprop="url"           (value: full canonical URL)
   1 itemprop="image"         (value: image URL on aurapharma.org)
   1 itemprop="offers"
   1 itemprop="price"         (value: "1.0")
   1 itemprop="priceCurrency" (value: "QAR")
   1 itemprop="listPrice"

What is populated: name, url, image, offers (with price + priceCurrency), listPrice.
What is missing: description, sku, brand, availability, itemCondition, aggregateRating / review (when reviews exist), mpn or gtin.

Shop page (/shop)

$ curl -s "https://www.aurapharma.org/shop" \
    | grep -oiE 'itemtype="[^"]+"' | sort | uniq -c

   8 itemtype="http://schema.org/Product"
   8 itemtype="http://schema.org/Offer"

Each product card on the listing emits its own Product + Offer, so the page declares itself as eight separate products. The category page is not eight products. It is a list of pointers to eight products. The right schema is a single ItemList whose itemListElements are ListItem nodes pointing at each product URL.

Zero structured data on the homepage and on every static page. No Organization, no WebSite, no LocalBusiness, no FAQPage, no BreadcrumbList.

What needs to change

1. Migrate the existing product microdata to JSON-LD

Microdata still works, but Google's current documentation uses JSON-LD in every example, the format is easier to maintain (one block at the bottom of the page rather than attributes scattered across the markup), and decoupling structured data from presentation HTML makes future template changes safer. Migrate the existing Product + Offer microdata on the 9 product pages to JSON-LD, and at the same time fill the missing fields.

2. Replace the shop-page Product blocks with a single ItemList

Stop emitting Product microdata on category cards. The shop page is a list, so it should emit ItemList with each card as a ListItem. The Product microdata still belongs on the actual product detail pages.

3. Add missing schema types

  • Sitewide: Organization on every page (in the base template).
  • Homepage: WebSite with SearchAction (only ship if the search endpoint actually works).
  • Contact: LocalBusiness with the Qatar address.
  • FAQ: FAQPage with the Q and A pairs that already exist on the page.
  • All non-home pages, once breadcrumbs ship (see master sheet row 46): BreadcrumbList.

4. Fix the protocol

Use https://schema.org in the @context field, not http://schema.org.

Do not ship Product schema until pricing is real. The current microdata exposes price="1.0", priceCurrency="QAR" on every product. That is the placeholder value flagged in master sheet row 92. If we migrate to JSON-LD without first fixing the prices, Google indexes 1.0 QAR for everything. Either set real prices first, or temporarily strip the offers block from the JSON-LD until prices are real.

Reference JSON-LD blocks

Treat each block as a template. Replace {{...}} placeholders with real values at render time. Output one combined <script type="application/ld+json"> array per page or multiple separate scripts. Both are valid.

Organization (sitewide, base template)

{
  "@context": "https://schema.org",
  "@type": "Organization",
  "@id": "https://www.aurapharma.org/#organization",
  "name": "Aura Pharma",
  "url": "https://www.aurapharma.org/",
  "logo": "https://www.aurapharma.org/web/image/website/1/logo",
  "description": "Aura Pharma is a leading company in non-pharmacological and cosmeceutical products in Qatar.",
  "address": {
    "@type": "PostalAddress",
    "addressCountry": "QA",
    "addressLocality": "{{city}}",
    "streetAddress": "{{street}}",
    "postalCode": "{{postal_code}}"
  },
  "contactPoint": [{
    "@type": "ContactPoint",
    "telephone": "{{phone_e164}}",
    "contactType": "customer service",
    "email": "{{contact_email}}",
    "areaServed": "QA",
    "availableLanguage": ["en", "ar"]
  }],
  "sameAs": [
    "{{facebook_url}}",
    "{{instagram_url}}",
    "{{linkedin_url}}",
    "{{twitter_url}}"
  ]
}

The logo field above uses aurapharma.org. The current OG image still points at aurapharma.odoo.com; both should be migrated to the production domain at the same time.

WebSite + SearchAction (homepage only)

{
  "@context": "https://schema.org",
  "@type": "WebSite",
  "@id": "https://www.aurapharma.org/#website",
  "url": "https://www.aurapharma.org/",
  "name": "Aura Pharma",
  "publisher": { "@id": "https://www.aurapharma.org/#organization" },
  "inLanguage": "en",
  "potentialAction": {
    "@type": "SearchAction",
    "target": {
      "@type": "EntryPoint",
      "urlTemplate": "https://www.aurapharma.org/website/search?search={search_term_string}"
    },
    "query-input": "required name=search_term_string"
  }
}

Only ship the SearchAction after AURA-006 (noindex on internal search results) has shipped. The search endpoint will still return results to users; it just stops being indexable. The SearchAction declaration is fine alongside that.

Product + Offer (every product page, replaces existing microdata)

  "image": [

    "@type": "Brand",

  "offers": {

Once this ships, remove the existing itemscope / itemtype / itemprop attributes from the product detail template. Don't run both formats in parallel; pick one. JSON-LD is the recommended one.

Implementation order

  1. Ship Organization first in the base template. Lowest risk, unlocks brand-entity recognition for every page.
  2. Replace shop microdata with ItemList on /shop. Stops the wrong-schema signal Google is currently receiving.
  3. Migrate product microdata to JSON-LD on the 9 product pages. Hold this until the placeholder 1.0 QAR pricing is replaced with real prices, or strip the offers block until pricing is real.
  4. Ship FAQPage as soon as the FAQ content is finalised. One of the fastest paths to a SERP rich result.
  5. Ship LocalBusiness on Contact once the Qatar office address, phone number, and hours are confirmed.
  6. Ship BreadcrumbList alongside the breadcrumb UI work in master sheet row 46.

Verification

  • Validate every template with the Google Rich Results Test. Expected: zero errors, warnings acknowledged.
  • Validate with the Schema.org Validator. Expected: full schema-spec compliance.
  • In DevTools: document.querySelectorAll('script[type="application/ld+json"]').length. Expected: at least 1 on every page.
  • Confirm the old itemtype and itemprop attributes are removed from product and shop templates after JSON-LD ships. Running both formats in parallel can confuse parsers.
  • Once Search Console is connected, confirm Enhancements starts surfacing Organization, Product, FAQ, Breadcrumb, Sitelinks searchbox reports without errors.
  • Search site:aurapharma.org in Google after deploy. Expected: rich results begin appearing for FAQ and product entries.

Why this is critical

Aura is a small new brand competing against entrenched retailers and aggregators. Structured data is one of the few SEO levers that does not require backlinks or domain authority to work. Half the pages on the site already have schema, just not the right schema in the right format with the right fields. Closing those gaps and migrating to JSON-LD makes every other ticket pay off harder, because Google can finally read what the site is about without guessing.
