Issue

The most valuable URL on the site is non-indexable. Loading https://www.aurapharma.org/ returns a page whose canonical link points to a different URL:

<link rel="canonical" href="https://www.aurapharma.org/home"/>

This tells search engines to drop / from their index and credit /home instead. To compound the problem, the XML sitemap lists both URLs as separate entries, which sends a contradictory signal:

<url><loc>https://www.aurapharma.org/</loc></url>
<url><loc>https://www.aurapharma.org/home</loc></url>

The Open Graph tags also point at /home, so social shares of the root URL also surface the wrong canonical. Most internal navigation links across the site point to /, which means every internal link is currently passing equity to a non-canonical URL.

Root cause

The CMS is configured to treat /home as the canonical homepage path while still serving the same content at /. The default sitemap generator picks up both URLs because both return HTTP 200. The previous migration from the old subdomain did not consolidate to a single homepage path.

Note that this same canonical issue is mirrored on the legacy aurapharma.odoo.com subdomain, where the canonical also points to /home. Fixing the canonical on the production host will not propagate to the legacy subdomain automatically.

Fix

Decide which URL is the canonical homepage. Recommendation: /, because (a) it is the URL most external sites and users will link to, (b) it is shorter, and (c) the HTTP redirect from the apex domain already lands on /.

Required changes:

  1. On the homepage template, replace the canonical so it points to itself: <link rel="canonical" href="https://www.aurapharma.org/">.
  2. Update Open Graph and Twitter meta on the homepage so og:url matches the new canonical.
  3. Add a server-level 301 redirect from /home to /.
  4. Regenerate the XML sitemap so it lists only valid canonical URLs returning 200. Specifically:
    • Keep /
    • Remove /home
    • Remove /website/info (the CMS "powered by" page; it should also be added to robots.txt Disallow)
    • Refresh the <lastmod> dates on all entries
  5. If the legacy aurapharma.odoo.com subdomain is still controllable, update its canonical to point to / as well. If not, leave it (it is already blocked via robots.txt Disallow).
Do not rewrite all the internal navigation links from / to /home as a workaround. That is treating the symptom, not the cause, and it will need to be reverted later.

Verification

  • Run curl -s https://www.aurapharma.org/ | grep -i canonical. Expected output: a single link rel="canonical" pointing to https://www.aurapharma.org/.
  • Run curl -sI https://www.aurapharma.org/home. Expected: HTTP 301 with Location: https://www.aurapharma.org/.
  • Run curl -s https://www.aurapharma.org/sitemap.xml | grep -E '/home|/website/info'. Expected: no matches.
  • Open https://www.aurapharma.org/ in a browser, view source, confirm og:url matches the canonical.
  • Once Google Search Console is connected, use Inspect URL on / and confirm the Google-selected canonical equals the user-declared canonical.
  • Resubmit the sitemap in Search Console after the fix is deployed.

Why this is critical

The root URL is the page that accumulates the most external backlinks, the most direct-traffic visits, and the most internal link equity. Letting it canonicalise away to a duplicate means none of that signal compounds where it should. Combined with the sitemap conflict, this is the single highest-leverage fix on the site and it should ship before any content or paid-media work begins.
