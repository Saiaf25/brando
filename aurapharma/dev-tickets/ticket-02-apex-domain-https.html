Issue

Visitors who type aurapharma.org (without www) hit a redirect chain that crosses two unrelated servers and drops to HTTP before being upgraded back to HTTPS. The current chain is:

https://aurapharma.org/
  ↓ 301 (server: Squarespace)
http://www.aurapharma.org/
  ↓ 301 (server: Odoo.sh)
https://www.aurapharma.org/

Two distinct problems are visible here:

  1. Wrong host. The apex domain is being served by Squarespace, not by the same hosting layer as the rest of the site. The HTTP response includes server: Squarespace and a Squarespace-specific x-contextid header. This means the apex DNS records are pointing at a Squarespace parking page, probably left over from a previous registrar configuration.
  2. Wrong scheme. The first redirect lands on http://www.aurapharma.org, not https://www.aurapharma.org. There is a brief window during which the user is on plain HTTP before HSTS upgrades them. Modern browsers will auto-upgrade the second hop, but the published 301 still says Location: http://..., which is what crawlers will record.

Verified evidence

$ curl -sI https://aurapharma.org/
HTTP/2 301
location: http://www.aurapharma.org
server: Squarespace
x-contextid: UWxVUt9F/JJbga3gz

$ curl -sI http://www.aurapharma.org/
HTTP/1.1 301 Moved Permanently
Server: nginx
Location: https://www.aurapharma.org/
Server: Odoo.sh

Fix

The fix is at the DNS layer, not in the CMS. Two changes are needed:

  1. Repoint the apex DNS records. Update the A and AAAA records (or ALIAS / ANAME if your DNS provider supports them) for aurapharma.org so they point at the same hosting layer that already serves www.aurapharma.org. The current Squarespace association needs to be removed.
  2. Configure a single 301 redirect at that hosting layer from https://aurapharma.org/ directly to https://www.aurapharma.org/, preserving the path. Confirm the redirect is HTTPS-to-HTTPS, not HTTP-to-HTTP.

The decision about whether the canonical host is www or non-www has already been made implicitly: www is the canonical (HSTS is set on it, the SSL certificate is valid for it, the sitemap and canonicals all use it). Do not change that. Just fix the apex so it consolidates to www in one hop.

The http://www.aurapharma.org → https://www.aurapharma.org redirect is already working correctly on the Odoo.sh layer. Keep it. The fix is only for the apex.

Verification

  • Run curl -sI https://aurapharma.org/. Expected: a single 301 with Location: https://www.aurapharma.org/ and server header matching the production host (not Squarespace).
  • Run curl -sIL https://aurapharma.org/about. Expected: a single redirect, then a 200 from the production host.
  • Visit https://aurapharma.org/ in a browser with developer tools open. Expected: one redirect entry in the network panel, no HTTP entries.
  • Test SSL on the apex: ssllabs.com. Expected: certificate valid for both aurapharma.org and www.aurapharma.org.
  • Confirm strict-transport-security header is present on the apex response too, so users who type the bare domain get HSTS protection.

Why this is critical

Users routinely type the bare domain. Crawlers do too. Today every one of those hits passes through a server (Squarespace) that has nothing to do with the brand, on plain HTTP. That is a security signal Google notices, a small but real exposure window for users on hostile networks, and a confusing data point if anyone ever audits this site externally. The fix is a 15-minute DNS change but it removes a class of issues no other ticket can address.
